# 热重载与滚动重建最佳实践

本文档面向 `new-api` 这类 API 中转网关场景，重点解决一个现实问题：

- 我们需要重建前端和后端
- 但用户正在进行流式请求、SSE 对话、长连接 AI 生成
- 重启容器时，正在进行中的请求可能被中断

因此，本文把“热重载”拆成两个层次来说明：

- 单实例的快速替换方案
- 面向流式请求的生产级滚动重建方案

## 1. 先说结论

### 1.1 当前仓库默认可直接执行的方案

如果你当前还是单实例部署，最佳实践是：

```bash
docker compose build new-api
docker compose up -d --no-deps new-api
docker compose ps
curl http://127.0.0.1:18900/api/status
```

这个方案的优点：

- 只重建 `new-api` 应用容器
- 不重建 `postgres` / `mysql` / `redis`
- 不改端口映射
- 不改 Cloudflare Tunnel
- 前端和后端会一起更新，因为当前 `Dockerfile` 会把前端静态资源打进同一个镜像

这个方案的边界：

- 它是“快速替换”
- 不是严格意义上的“无损热重载”
- 如果容器切换时有正在进行中的流式响应，旧连接仍然可能被中断

### 1.2 生产环境推荐的最佳实践

如果你的目标是尽量不影响流式请求和 AI 对话，推荐方案是：

- 保持 Cloudflare Tunnel 指向一个固定本地入口
- 这个固定入口不直接绑定某一个 `new-api` 容器
- 在本地固定入口前面增加一个反向代理
- 后面跑两个 `new-api` 应用实例，做滚动切换

推荐拓扑：

```text
Cloudflare Tunnel
        |
        v
127.0.0.1:18900  (固定入口，不变)
        |
        v
本地反向代理（Nginx / Caddy / HAProxy）
      /   \
     v     v
new-api-blue   new-api-green
       \       /
        v     v
     PostgreSQL/MySQL + Redis
```

在这个模式下：

- Tunnel 配置不需要跟着容器切换
- 对外端口不需要改
- 新版本实例先启动、先健康检查
- 代理只把新流量切到新实例
- 旧实例继续把已建立的流式请求跑完

这才是最接近“不中断流式请求”的最佳实践

## 2. 为什么不能直接 `docker compose down && up -d`

对于 `new-api` 这种 API 中转网关，直接整套 `down` 再 `up` 有几个明显问题：

- 当前进行中的 SSE / 流式请求会被直接断开
- 与上游 AI 的长连接会中断
- 用户正在生成中的回答会失败或半截结束
- 如果同时把 Redis 也重启，缓存、会话、限流状态会一起抖动
- 如果同时把数据库容器也重启，应用恢复时间会更长

所以，生产环境不推荐：

```bash
docker compose down
docker compose up -d
```

也不推荐：

```bash
docker compose down && docker compose up -d
```

## 3. 当前仓库的特点

在制定最佳实践前，需要先明确当前仓库的几个事实：

### 3.1 当前 `Dockerfile` 是前后端同镜像

当前 [Dockerfile](/home/dev/new-api/Dockerfile) 会：

- 先构建 `web/` 前端产物
- 再把前端 `dist` 打进 Go 二进制镜像

这意味着：

- 一次镜像重建，前端和后端会一起更新
- 不需要单独再为前端做一套容器切换

### 3.2 当前应用没有显式优雅停机流程

当前 [main.go](/home/dev/new-api/main.go) 直接使用 `server.Run(...)` 启动服务，没有显式 `Shutdown()` 流程。

这意味着：

- 单实例容器被停止时，不能指望应用自己优雅排空所有流式连接
- 想要尽量不影响用户请求，不能只依赖容器停机宽限时间
- 真正有效的手段是在应用前面做代理排空和滚动切换

### 3.3 单实例和双实例适用的数据库策略不同

如果你使用：

- SQLite：适合单实例快速替换，不适合双实例滚动重建
- PostgreSQL / MySQL：适合双实例滚动重建

原因很直接：

- 双实例共享同一个 SQLite 文件会带来锁竞争和一致性风险
- 滚动重建推荐使用 PostgreSQL 或 MySQL

## 4. 方案 A：单实例快速替换

这是当前仓库默认最容易执行的方案，也是“改动最少”的方案。

### 4.1 适用场景

- 当前只有一个 `new-api` 应用实例
- Cloudflare Tunnel 直接指向当前应用监听端口
- 你想保持现有部署结构不变
- 你能接受极短时间的切换中断
- 你当前使用 SQLite，或者暂时不准备改成双实例架构

### 4.2 标准命令

```bash
docker compose build new-api
docker compose up -d --no-deps new-api
docker compose ps
curl http://127.0.0.1:18900/api/status
```

### 4.3 关键原则

- 先 `build`，再 `up -d`
- 只替换 `new-api`
- 不要重建 `postgres`
- 不要重建 `redis`
- 不要改 Cloudflare Tunnel
- 不要改对外端口

### 4.4 这种方案的真实效果

优点：

- 重建速度快
- 操作简单
- 数据库和缓存不会跟着一起重启
- 本地端口和 Tunnel 目标保持不变

缺点：

- 容器替换瞬间，仍然可能中断进行中的流式请求
- 对普通短请求影响通常很小
- 对长时间生成、长对话、长流式输出不够友好

因此，它更准确的说法应该是：

- “快速替换”
- 不是“无损热重载”

## 5. 方案 B：流式请求友好的滚动重建

这是 `new-api` 作为 API 中转件时更推荐的长期最佳实践。

### 5.1 核心原则

- Cloudflare Tunnel 只面向一个固定本地入口
- 固定入口由反向代理持有，不由某个应用容器直接持有
- 后面同时保留两个应用实例
- 发布新版本时，不直接杀掉旧实例
- 先让新实例上线，再把新流量切过去
- 旧实例只负责把已有连接跑完

### 5.2 推荐架构

固定入口：

- `127.0.0.1:18900`

角色划分：

- Cloudflare Tunnel：保持现有映射，不需要改公网配置
- 本地反向代理：绑定 `127.0.0.1:18900`
- `new-api-blue` / `new-api-green`：应用实例
- PostgreSQL / MySQL：主数据库
- Redis：共享缓存、限流、会话相关能力

### 5.3 必要前提

要做双实例滚动重建，建议满足下面条件：

- 使用 PostgreSQL 或 MySQL，不使用 SQLite
- 两个应用实例共享同一个 Redis
- 两个应用实例的 `SESSION_SECRET` 完全一致
- 如果启用了 Redis 加密，`CRYPTO_SECRET` 也必须完全一致
- 两个应用实例使用同一套业务配置源

### 5.4 标准发布流程

推荐发布顺序如下：

1. 先构建新镜像，不动线上实例

```bash
docker compose build new-api
```

2. 启动“待切换”的新实例

- 例如当前线上是 `new-api-blue`
- 那么先把 `new-api-green` 用新镜像拉起来

3. 等待新实例健康检查通过

至少检查：

- 容器健康状态
- `/api/status`
- 必要的鉴权与数据库连通

4. 让代理开始把新请求分配给新实例

- 可以先双活
- 也可以直接让新流量只进新实例

5. 让旧实例进入排空状态

- 旧实例不再接收新流量
- 但保留已有连接

6. 等待排空窗口结束，再下掉旧实例

建议：

- 排空窗口至少不要小于 `STREAMING_TIMEOUT`
- 对话和长流式场景建议给 5 到 15 分钟的排空窗口
- 如果你的用户经常跑长回答、长推理或大输出，排空时间要更长

7. 旧实例完全排空后，再停止旧实例

### 5.5 为什么这个方案更适合流式请求

原因很简单：

- 新连接已经切到新实例
- 旧连接还保留在旧实例
- 代理的平滑重载不会立刻中断旧连接
- 用户长对话可以自然结束

真正被中断的概率会显著低于“直接替换单实例”

## 6. Cloudflare Tunnel 的建议

如果你已经通过 Cloudflare Tunnel 对外暴露服务，推荐原则是：

- Tunnel 的公网域名不要跟着版本切换
- Tunnel 的本地目标地址不要跟着应用容器切换

最佳实践是：

- Tunnel 永远指向一个固定本地端口
- 固定本地端口由反向代理占用
- 应用实例在代理后面滚动切换

这样做的好处：

- 公网配置不需要动
- 本地端口不需要动
- 发布动作只发生在代理后面的应用层

## 7. 配置建议

### 7.1 单实例最少应该做到

- 使用 `docker compose build new-api`
- 使用 `docker compose up -d --no-deps new-api`
- 不要 `docker compose down`
- 不要一起重建数据库和 Redis

### 7.2 双实例至少应该做到

- 使用 PostgreSQL / MySQL
- 使用共享 Redis
- 统一 `SESSION_SECRET`
- 统一 `CRYPTO_SECRET`
- 使用固定代理入口
- 应用实例不直接承接 Tunnel 的固定端口

### 7.3 与流式请求相关的参数建议

建议关注：

- `STREAMING_TIMEOUT`
- 代理层的超时设置
- 代理层的平滑重载能力

如果你经常承载长流式输出：

- `STREAMING_TIMEOUT` 不要设置得过短
- 排空窗口不要短于流式超时

## 8. 明确不推荐的做法

不推荐：

- `docker compose down && docker compose up -d`
- 发布时同时重建 `new-api`、`postgres`、`redis`
- 双实例共享 SQLite 数据库文件
- 让 Cloudflare Tunnel 直接跟着某一个应用容器做切换
- 把“单实例快速替换”误认为“不会中断用户流式请求”

## 9. 当前项目的最佳实践结论

### 9.1 如果你现在是单实例

当前项目的默认最佳实践是：

```bash
docker compose build new-api
docker compose up -d --no-deps new-api
docker compose ps
curl http://127.0.0.1:18900/api/status
```

这套流程适合作为日常更新方式。

### 9.2 如果你追求“尽量不影响流式对话”

当前项目更推荐的长期最佳实践是：

- 保持 Cloudflare Tunnel 外部入口不变
- 保持本地固定端口不变
- 在固定入口前使用本地反向代理
- 后面跑 `blue/green` 两个 `new-api` 实例
- 用 PostgreSQL / MySQL + Redis
- 采用滚动切换和代理排空

换句话说：

- 单实例快速替换，适合“少改动、可接受短中断”
- 双实例滚动切换，适合“API 网关、流式请求优先、尽量不打断用户”

## 10. 验收清单

发布前检查：

- 已提前完成镜像构建
- 数据库和 Redis 不会在本次应用发布中被重建
- Cloudflare Tunnel 配置保持不变
- 对外端口保持不变
- `/api/status` 能在切换后正常返回

如果你使用双实例：

- 新实例已经健康
- 代理已经完成平滑切换
- 旧实例已停止接收新流量
- 旧实例已留够排空时间

## 11. 一句话总结

对于 `new-api` 这种 API 中转网关：

- **单实例最优动作是“先构建，再只替换应用容器”**
- **真正对流式请求友好的最佳实践，是“固定 Tunnel 入口 + 本地反向代理 + 双实例滚动切换”**
